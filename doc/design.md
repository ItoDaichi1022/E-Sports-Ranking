# トーナメント運営＆ランキングサイト 設計ドキュメント

---

## 1. 目的

- 自作サイト内で **トーナメントの運営**（募集・ブラケット生成・勝敗入力）と **個人ランキング** を完結させる
- 選手が**自分のアカウント**を持ち、自分のプロフィールを自分で編集できる
- 無料・低メンテナンスで運用する

## 2. 想定規模

- 1大会あたり 25〜60人程度
- 1シーズンで数百試合程度
- 全データを合わせても数MB規模 → ブラウザ内で全読み込み・全計算が可能

## 3. 技術構成

| 役割 | 使うもの | 備考 |
|------|----------|------|
| ホスティング | GitHub Pages | 無料・無期限。ビルド工程なし |
| データベース | Supabase (PostgreSQL) | 無料枠500MB |
| 認証 | Supabase Auth | Google / Discord / メール＋パスワード |
| アクセス制御 | 行レベルセキュリティ (RLS) | 閲覧は全員、書き込みは本人と運営に限定 |
| 自動更新 | Supabase Realtime | 変更が全端末へプッシュされる |
| ランキング計算 | ブラウザ内 JavaScript | サーバー計算不要 |

- **ビルド工程を持たない**方針は維持している。supabase-js は `js/vendor/supabase.js` に同梱し、
  CDNの障害でサイトが止まらないようにしている。
- セットアップ手順は [`supabase/SETUP.md`](../supabase/SETUP.md)、スキーマは [`supabase/schema.sql`](../supabase/schema.sql)。

### 以前の構成からの移行

当初はGitHubリポジトリ内のJSONファイルをデータベース代わりに使い、共有の書き込みトークンで
認証していた。ユーザーアカウント・本人だけが編集できる制御・エントリーの同時登録は
この方式では安全に実現できないため、Supabaseへ移行した。

移行によって不要になったもの：

- 楽観ロックとレコード単位の三方マージ（約250行）。Postgresは行単位で書き込むため、
  別々の大会・別々の選手を同時に編集しても競合しない
- コミットSHAを比較する10秒ポーリング。Realtimeの購読に置き換えた
- 変更をまとめてコミットする自動保存デバウンス。各操作がその場で書き込まれる

旧データの移行は `scripts/migrate.mjs`、その検証は `scripts/verify-migration.mjs` が行う。

## 4. 権限モデル

| | 閲覧 | 自分のプロフィール編集 | エントリー | 大会運営 |
|---|:---:|:---:|:---:|:---:|
| ゲスト（未ログイン） | ○ | – | – | – |
| ログイン済み（選手） | ○ | ○ | ○ | – |
| 運営（`role = 'admin'`） | ○ | ○（全選手） | ○ | ○ |

- **閲覧は認証不要**。大会履歴もブラケットも選手プロフィールもログインせずに見られる。
- 制御はすべてデータベース側のRLSで行う。画面上のボタンの出し分けは利便性のためのもので、
  防御ではない（APIを直接叩かれても通らない）。
- 一般ユーザーが自分を運営に昇格させられないよう、`role` と `user_id` は
  **列単位のGRANTでUPDATE不可**にしてある。変更は `admin_set_player_role` などのRPC経由でのみ可能。
- RLSポリシーの中から `players` を参照すると無限再帰になるため、
  管理者判定は `security definer` 関数 `is_admin()` に切り出している。

## 5. 選手の識別設計

- **選手 ＝ ユーザーアカウント**。`players.user_id` が `auth.users` を指す。
  新規の選手は自分で登録し、その瞬間から自分のプロフィールを編集できる（運営の承認は不要）。
- `user_id` が null の行は**運営が代理登録した選手**。旧データから移行してきた選手や、
  アカウントを持たない参加者がこれにあたる。本人がログインしてきたら、
  運営が選手一覧から `admin_merge_players` で対応付ける（初回だけの作業）。
- **不変の一意キーは `players.id`（uuid）**。試合結果もランキングもこのIDに紐づく。
- **表示名は可変**。変更すると旧名が `past_names` に残るので、名前が変わっても戦績が分断されない。
- **ゲームアカウントIDは本人が自由に記入・変更できる表示用の欄**。以前は主キーだったが、
  編集可能にするにあたり主キーから外した（移行時に旧IDは `game_account_id` へ引き継いでいる）。

## 6. データベース設計

テーブル定義の実体は [`supabase/schema.sql`](../supabase/schema.sql)。要点だけ記す。

| テーブル | 役割 |
|---|---|
| `players` | 選手＝アカウント。プロフィール（自己紹介・使用キャラ・SNS・ゲームID）と `role` を持つ |
| `tournaments` | 大会。`status` でライフサイクルを表す。`capacity` は定員 |
| `tournament_entries` | エントリー。締切後に `seed` が入る。主キーは (大会, 選手) |
| `brackets` | ブラケット1件をJSONBで保持 |
| `matches` | 確定した試合。ランキング計算の唯一の入力 |
| `published_rankings` | 「公開」した瞬間のランキングのスナップショット |

- **ブラケットだけはJSONB**にしている。`js/bracket.js` が扱うオブジェクト構造をそのまま保存でき、
  ブラケットの生成・進行ロジックを一切書き換えずに移行できたため。
  ブラケットは常に1大会分をまとめて読み書きするので、正規化する利点も薄い。
- **常時計算するランキングスコアはどこにも保存しない**。運営の画面では常に `matches` から再計算する。
  例外として、明示的に「公開」した瞬間のランキングだけを `published_rankings` に1行追加する
  （上書きではないので、いつ何を公開したかの履歴が残る）。
- 集計ロジックを変えても過去データから作り直せて、データ不整合が起きない。

### in-memory state との対応

`js/db.js` がデータベースとアプリの唯一の境界で、snake_case ⇄ camelCase の変換もここだけで行う。
そのため `js/ranking.js`・`js/bracket.js`・`js/playerStats.js` などの計算・描画ロジックは
ストレージを一切意識しない（`js/state.js` の形だけを見て動く）。

`tournaments.participantIds` は `tournament_entries` をシード順（未確定なら登録順）に
並べたものとして組み立てられる。ブラケットのシード番号表示もこの並びを使う。

### published_rankings のスナップショット形式

```json
{
  "published_at": "2026-07-19T12:34:56.000Z",
  "period_months": 3,
  "data": {
    "rankings": [
      { "id": "uuid", "name": "Gyu", "score": 100, "tournamentsPlayed": 4,
        "rank": 1, "previousRank": 2,
        "bestAchievement": { "label": "優勝", "tournamentName": "○○杯",
                             "participantCount": 32, "tier": "Tier 4", "value": 32 } }
    ]
  }
}
```

- `previousRank`：公開した瞬間の「ひとつ前に公開されていたランキング」での順位。
  前回に存在しなかった選手は `null`（NEW扱い）。公開時に一度だけ計算して焼き込むため、
  以後 `matches` が変わっても遡って変化しない。
- `bestAchievement`：集計期間内の出場大会から、参加人数÷勝ち上がり深さ（優勝=1、準優勝=2、
  ベストN=N）が最大の成績を1つ選出したもの。ランキング発表カード画像（PNG書き出し）で使用する。
- 閲覧者のランキングページ・選手個人ページの「現在ランク」は、常に最新のスナップショットを表示する。

## 7. 大会のライフサイクル

```
draft ──募集開始──▶ recruiting ──締切──▶ running ──決勝確定──▶ finished
（準備中）           （募集中）           （進行中）          （終了）
```

- **draft** — 運営が大会名・日付・ルール・定員を作成した状態。募集ページには運営にだけ見える
- **recruiting** — 募集ページに掲載される。ログイン済みユーザーは「エントリー」ボタン1つで参加登録でき、
  募集中はいつでも取り消せる
- **締切時** — エントリー済み選手を**その時点のランキング順に自動整列**してシードを確定し、
  ブラケットを生成する（`js/entries.js` の `seedByRanking`）。ランキングに載っていない選手は
  後ろにまとめ、その中では登録順を保つ
- **running** — 勝敗入力。確定のたびに `matches` へ記録され、各選手のプロフィールの戦歴に即座に反映される
- **finished** — 決勝が確定すると自動で遷移する

運営が募集を使わず参加者を直接選んで大会を立てる経路も残してある（大会作成画面の
「運営が参加者を直接選ぶ」）。この場合は作成と同時に `running` になる。

### 定員の扱い

定員の判定は**データベースのトリガー**で行う。クライアント側で残枠を確認してからINSERTする方式は、
2人が同時にボタンを押したときに両方とも通ってしまうため。大会行を `FOR UPDATE` でロックしてから
数えることで、同じ大会へのエントリーを直列化している。

## 8. ランキング方式（LumiRank軽量版）

### 背景

本家LumiRankは、互いに直接対戦しない世界中の選手を比較するため、Tier別ポイント表・地域ポイント・
反復averaging等の重い仕組みを持つ。本ケースは同一コミュニティ内で25〜60人が繰り返し対戦するため
対戦グラフが密で、重い仕組みは不要。**核（相手の強さで重み付けした反復スコアリング）だけを残す**。

### 残す要素

- 相手の強さで重み付けした勝敗スコア
- スコアが収束するまでの反復計算（密なグラフなら数回で収束）
- 大会規模による重み付け

### 省く要素

- Tier別ポイント表 / 地域ポイント / superregion（全大会が同一コミュニティ内のため不要）
- Hidden Boss値（不要）
- 出場ペナルティは「ランク付け対象は最低N大会出場」の足切りに簡略化

### アルゴリズム（最小形）

1. 各選手のスコア S を全員同じ初期値で開始
2. 1試合ごとに、勝者は敗者のスコアに応じたポイントを得る（強い相手ほど高い）／敗者は失う。大会規模で重み付け
3. 全試合を集計して各選手のスコアを更新
4. 更新後のスコアで再度全試合を計算 → これを数回繰り返しスコアが収束したら確定
5. 最後に #1 が 100 になるようスケーリング
6. 最低出場数に満たない選手はランク付け対象外

### 確定パラメータ

- **スケーリング上限**：#1 = 100
- **足切り大会数 N**：1（1大会以上の出場でランク付け対象）
- **ブラケットのシード配置**：現在のランキング順を元にシードする（上位が早期に当たらないよう配置）

## 9. 大会規模Tier（表示用ラベル）

コミュニティの慣習に合わせ、参加人数から `js/tournamentTier.js` が自動でTierを判定する（保存はせず、都度算出）。
10人刻みで Tier 1（1〜9人）〜 Tier 6（50〜59人）、**60人以上は「大規模」**判定。
ランキング計算の `weight` とは無関係の独立した区分で、大会情報・大会履歴一覧・
ランキング発表カード画像に表示する。

## 10. トーナメント形式の実装順

1. **シングルエリミネーション** ← 実装済み
2. **ダブルエリミネーション**：敗者ブラケットを追加
3. **総当たり（リーグ）**：全組み合わせ生成

`tournaments.format` に `double_elim` / `round_robin` を入れられるようスキーマ側は用意してある。
