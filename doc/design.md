# トーナメント運営＆ランキングサイト 設計ドキュメント

このドキュメントは Claude Code に渡して実装を進めるための設計仕様です。

---

## 1. 目的

- 自作サイト内で **トーナメントの運営**（ブラケット生成・勝敗入力）と **個人ランキング** を完結させる
- 無料・無期限・低メンテナンスで運用する（データベースサーバーを持たない）
- 複数人の運営者が、複数端末から操作できる

## 2. 想定規模

- 1大会あたり 25〜60人程度
- 1シーズンで数百試合程度
- 全データを合わせても数十KB規模 → ブラウザ内で全読み込み・全計算が可能

## 3. 技術構成（サーバーレス方針）

| 役割 | 使うもの | 備考 |
|------|----------|------|
| ホスティング | GitHub Pages | 無料・無期限・pauseなし |
| データ保存 | GitHubリポジトリ内のJSONファイル | 履歴＝バックアップ兼監査ログ |
| データ読み書き | GitHub API | 運営者のみ書き込み。閲覧者は認証不要 |
| ランキング計算 | ブラウザ内 JavaScript | サーバー計算不要 |

- サイトのコードとデータJSONは **同一リポジトリに同居** させる。
- リポジトリ1個をデプロイすれば全運営者が同じ状態を共有できる。
- DBサーバーは不要。将来書き込み頻度が上がった場合のみ Cloudflare D1 等への移行を検討（同じサーバーレス路線）。

## 4. 認証方針（書き込み時のみ）

- **閲覧（ランキング/ブラケット表示）**：認証不要。公開URLを見るだけ。
- **書き込み（大会作成・勝敗入力）**：GitHubへの書き込み権限が必要。
  - 方針A：運営者各自のGitHubアカウントをリポジトリのコラボレーターに追加し、個人トークンで認証（誰の操作かがコミット履歴に残る／推奨）
  - 方針B：運営チームで共有の書き込みトークンを1つ用意（GitHubアカウント不要だが操作者の区別がつきにくい）
- ※認証はロードマップの後半で実装。まずスキーマとブラケットから着手する。

## 5. 選手の識別設計

- **ゲームアカウントID = 不変の一意キー**。全選手が保有しユニーク。試合結果もランキングもこのIDに紐づける。
- **選手名 = 可変の表示ラベル**。名前変更時は現在名を書き換え、旧名を保持する。
- これにより名前が変わっても戦績が分断されない。

## 6. データスキーマ

### players.json
```json
{
  "players": [
    {
      "id": "SW-1234-5678-9012",
      "currentName": "Gyu",
      "pastNames": ["旧名A", "旧名B"]
    }
  ]
}
```
- `id`：ゲームアカウントID（不変キー）
- `currentName`：現在の表示名
- `pastNames`：過去の名前（任意・名前変更履歴）
- 過去に存在した `mainCharacters`（使用キャラ）は廃止済み。古いデータに残っていても無視される。

### tournaments.json
```json
{
  "tournaments": [
    {
      "id": "t001",
      "name": "大会名",
      "date": "2026-07-18",
      "format": "single_elim",
      "participantIds": ["SW-1234-5678-9012", "SW-9999-8888-7777"],
      "weight": null
    }
  ]
}
```
- `format`：`single_elim` / `double_elim` / `round_robin`
- `weight`：大会規模の重み。null の場合は参加人数から自動算出
- `participantIds`：参加選手のID配列

### matches.json
```json
{
  "matches": [
    {
      "id": "m001",
      "tournamentId": "t001",
      "winnerId": "SW-1234-5678-9012",
      "loserId": "SW-9999-8888-7777",
      "score": "2-1",
      "round": "WF"
    }
  ]
}
```
- ランキング計算はこのファイルが唯一の入力。
- 勝者・敗者はIDで記録（名前は保存しない）。

### bracket（大会ごと）
- ブラケット構造と各試合の確定/未確定状態を保持。
- 運営中の勝敗入力はここを更新し、確定した結果が matches に流れる。

### 設計原則
- **ランキングスコアはどこにも保存しない**。常に matches.json から再計算する。
- 集計ロジックを変えても過去データから作り直せて、データ不整合が起きない。

## 7. ランキング方式（LumiRank軽量版）

### 背景
本家LumiRankは、互いに直接対戦しない世界中の選手を比較するため、Tier別ポイント表・地域ポイント・反復averaging等の重い仕組みを持つ。本ケースは同一コミュニティ内で25〜60人が繰り返し対戦するため対戦グラフが密で、重い仕組みは不要。**核（相手の強さで重み付けした反復スコアリング）だけを残す**。

### 残す要素
- 相手の強さで重み付けした勝敗スコア
- スコアが収束するまでの反復計算（密なグラフなら数回で収束）
- 大会規模による重み付け

### 省く要素
- Tier別ポイント表 / 地域ポイント / superregion（全大会が同一コミュニティ内のため不要）
- Hidden Boss値（不要）
- 出場ペナルティは「ランク付け対象は最低N大会出場」の足切りに簡略化

### アルゴリズム（最小形）
1. 各選手のスコア S を全員同じ初期値で開始
2. 1試合ごとに、勝者は敗者のスコアに応じたポイントを得る（強い相手ほど高い）／敗者は失う。大会規模で重み付け
3. 全試合を集計して各選手のスコアを更新
4. 更新後のスコアで再度全試合を計算 → これを数回繰り返しスコアが収束したら確定
5. 最後に #1 が 100 になるようスケーリング
6. 最低出場数に満たない選手はランク付け対象外（または別枠表示）

## 8. トーナメント形式の実装順

「複数形式対応」が目標だが、段階実装する。

1. **シングルエリミネーション**（最優先）：ブラケット生成・勝敗入力・次ラウンド進出が最も素直
2. **ダブルエリミネーション**：敗者ブラケットを上に追加
3. **総当たり（リーグ）**：全組み合わせ生成

## 9. 実装ロードマップ（トークン節約のための分割）

1. データスキーマ確定 ← **完了（このドキュメント）**
2. シングルエリミネーションのブラケット生成 + 勝敗入力UI
3. LumiRank軽量版ランキング計算エンジン
4. GitHub API連携（保存・読込・複数人認証）
5. ダブルエリミ・総当たり追加

- **実装はこのチャットではなく Claude Code で進める**。ファイル単位の差分編集ができ、毎回コードを貼り直さずに済むためトークン効率が高い。
- 各ステップを個別に依頼し、一括実装は避ける。

## 10. 確定パラメータ

- **スケーリング上限**：#1 = 100
- **足切り大会数 N**：1（1大会以上の出場でランク付け対象）
- **認証方針**：方針B（運営チームで共有の書き込みトークンを1つ使用）
- **ブラケットのシード配置**：現在のランキング順を元にシードする（上位が早期に当たらないよう配置）
